API開発で「作業者に渡すべき情報」と「進め方（手順）」を、実務でそのまま使える粒度でまとめます。バックエンド（API）単体でも、フロントやモバイル・他社連携が絡んでも共通です。

1) 作業者に最初に渡すべき情報（最低限セット）

A. 目的・スコープ
	•	何を実現するAPIか（業務フロー / ユーザー価値）
	•	今回の範囲（MVPでやる/やらない、将来対応、Out of scope）
	•	成功条件（SLA/レイテンシ/同時接続/想定リクエスト数、ログ監査要件など）

B. 仕様（API仕様書の「確定版」）
	•	エンドポイント一覧（URL、HTTPメソッド）
	•	リクエスト/レスポンスのJSONスキーマ（必須/任意、型、例）
	•	ステータスコード規約（200/201/400/401/403/404/409/429/500…）
	•	エラー形式（共通フォーマット、エラーコード、メッセージ方針）
	•	認証・認可（JWT/OAuth/API Key、ロール、権限マトリクス）
	•	バージョニング方針（/v1、ヘッダ、後方互換ポリシー）
	•	互換性・破壊的変更のルール（deprecation方針）

C. データ設計
	•	DB種類、ER図、テーブル定義（主キー、ユニーク制約、外部キー、index）
	•	データのライフサイクル（作成/更新/削除、論理削除の有無）
	•	監査項目（created_by, updated_by, timestamps, request_id など）
	•	データ保持期間・削除要求（個人情報やログ保持）

D. 非機能要件（ここがないと炎上しがち）
	•	性能目標（例：p95 300ms、タイムアウト、最大ペイロード）
	•	レート制限・スロットリング（429の扱い）
	•	セキュリティ（OWASP観点、CORS、CSRF、入力検証、WAF有無）
	•	ログ/監視（ログ形式、メトリクス、アラート条件、トレーシング）
	•	可用性・冗長化・バックアップ（RPO/RTO）

E. 開発ルール・運用ルール
	•	リポジトリURL、ブランチ戦略（GitFlow / trunk-based）
	•	コーディング規約、lint/format、commit規約
	•	PR運用（レビュー観点、必須チェック、マージ条件）
	•	環境（local/dev/stg/prod）と差分（設定値、Secrets）
	•	デプロイ手順（CI/CD、手動手順が残る部分）

F. 連携仕様（外部・内部）
	•	連携先一覧（DB、Queue、S3、外部API、AI推論基盤など）
	•	リトライ方針・冪等性（idempotency key、重複排除）
	•	非同期処理の契約（Webhook、Job status API、通知）

⸻

2) 作業者に進めさせる手順（推奨の開発フロー）

Step 0: キックオフ（30〜60分で合意を作る）
	•	スコープと優先順位（Must/Should/Could）
	•	APIの契約を先に固める（OpenAPIの“契約ファースト”推奨）
	•	受け入れ条件（Definition of Done）を明文化

Step 1: 仕様の確定（ブレを止める）
	•	OpenAPI（Swagger）でエンドポイントを定義
	•	エラー形式・認証・ページングなど「共通規約」を先に決める
	•	例（サンプルリク/レス）を必ず付ける（フロントが実装できるレベル）

Step 2: 土台構築（後で効く）
	•	プロジェクト雛形、設定管理（.env、Secrets）、DI/ルータ構成
	•	DBマイグレーション導入
	•	ログ（構造化ログ）・request_id・例外ハンドリングの共通化
	•	CI（lint/test/build）を最初に通す

Step 3: 実装（薄く縦に切る）

おすすめは「縦割りで1本通す」：
	1.	認証（最小）
	2.	代表的な1エンドポイント（CRUDの核）
	3.	DB保存
	4.	テスト（ユニット＋API）
	5.	監視ログ確認
この1本が通ったら同じ型でエンドポイントを増やす

Step 4: テストと契約保証
	•	コントラクトテスト（OpenAPIからのバリデーション）
	•	エラー系（入力不正、権限なし、重複、外部API障害）
	•	負荷・性能（最低限：想定RPSでp95確認）
	•	セキュリティ（入力検証、認可抜け、シークレット漏洩）

Step 5: デプロイ〜運用の仕上げ
	•	dev/stg/prod に段階デプロイ
	•	監視アラートの閾値設定（5xx率、レイテンシ、キュー滞留）
	•	ロールバック手順とDBマイグレーションの戻し方
	•	Runbook（障害対応手順）を短くても作る

⸻

3) 作業指示をチケット化する時の「粒度」

各チケットには最低これを入れると、迷走が減ります。
	•	目的（誰の何のため）
	•	対象エンドポイント（OpenAPIの該当箇所リンク/抜粋）
	•	受け入れ条件（例：200/400/401/409の挙動、ログ出力、テスト追加）
	•	影響範囲（DB変更、既存API互換性）
	•	モック/サンプル（リクエスト例、レスポンス例）
	•	依存（先に必要なマイグレーション、外部APIキー等）

⸻

4) 最初に決めておくと強い「共通仕様テンプレ」
	•	認証：Authorization: Bearer <token>
	•	エラー形式：{ "error": { "code": "...", "message": "...", "details": [...] }, "request_id": "..." }
	•	ページング：limit, cursor（or offset）の統一
	•	冪等性：Idempotency-Key をPOSTに採用するか
	•	リクエスト追跡：X-Request-Id（なければサーバで発行）

⸻

必要なら、あなたのプロジェクト前提（例：Lambda + API Gateway、FastAPI、SageMaker推論、RDS/ DynamoDBなど）に合わせて、**「作業者に渡すパッケージ一式（OpenAPI雛形、DoD、チケット雛形、フォルダ構成、CIチェック項目）」**まで、コピペで使える形にして出します。
